# PRD-120 — SAML Identity Provider Configuration

---

## Layer Governance

This consolidated Layer 12 bundle follows the repo's manifest/catalog control plane:
- the module catalog and per-environment deployment manifest decide which identity capabilities are active
- the current target pack family for Layer 12 is `enterprise-identity`, not the older `identity-sso` placeholder
- `deployment_profile` only shapes runtime behavior and does not enable SSO on its own
- PRD-10 identity mode is a real creation-time prerequisite when SSO is intentionally selected
- optional alerting and audit sinks remain optional inputs, not universal dependencies

Governance posture: `Green`

## 3. CONTEXT & PURPOSE

Amazon Connect `identity_management_type` is immutable after instance creation. That fact governs the whole Layer 12 rollout.

If SSO is selected for an environment, the operator must:

1. enable the Layer 12 identity capability through the module catalog and environment deployment manifest
2. provision PRD-10 with the correct identity mode from the start
3. then apply PRD-120 and the rest of the Layer 12 bundle

This PRD establishes the IAM and SAML trust foundation between the organization identity provider and Amazon Connect. In the current repo shape, PRD-111 may host adjacent enterprise identity operations, but it must not be treated as the owner of the SAML organization instance or a hidden prerequisite for standalone deployments.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Manifest-Gated Activation

PRD-120 resources are created only when the identity capability is enabled in the environment manifest. Runtime tfvars may provide SAML metadata, but they do not decide module existence.

### FR-002 — SAML Provider Provisioning

Provision the IAM SAML provider, the federation role used for Amazon Connect sign-in, and the required assertion and relay-state mapping for Amazon Connect. The metadata document must be supplied explicitly through module inputs or a secure external source.

### FR-003 — Connect Identity-Mode Prerequisite

The PRD must clearly state that environments created as `CONNECT_MANAGED` cannot later adopt SAML in place. This is a rollout constraint, not a hidden implementation detail.

### FR-004 — Standalone And Multi-Account IdP Support

In multi-account environments, PRD-120 may rely on management-account-governed federation foundations or equivalent shared identity operations. In standalone environments, equivalent local federation is acceptable. The PRD must not imply that PRD-111 creates or owns the required SAML organization instance.

### FR-002 — Provisioning Adapter Surface

Provision a custom provisioning surface that can accept upstream identity events or SCIM-shaped requests and translate them to Amazon Connect user-management APIs such as `CreateUser`, `UpdateUserIdentityInfo`, `UpdateUserRoutingProfile`, `UpdateUserSecurityProfiles`, and `DeleteUser`. The module must remain viable even when PRD-50 and PRD-40 are absent.

### FR-003 — Optional Shared Foundations

If PRD-40 or PRD-50 are enabled, PRD-122 may integrate with them explicitly. Those are optional rollout choices, not hidden requirements.
