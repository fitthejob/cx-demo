## 2. CONTEXT & PURPOSE

The platform's PCI-DSS compliance posture covers data handling and network segmentation for a contact center that processes payment-adjacent information (account numbers, caller identity). It explicitly excludes verbal card capture — that is handled by a separate PCI-scoped system. This PRD hardens the enabled infrastructure to meet PCI-DSS requirements beyond what the baseline platform provides: a manual S3 Object Lock runbook for call recordings when an assessor requires immutability, enhanced KMS key policies, VPC deployment for Lambda (where required by assessor), and automated PCI control verification.

**Scope:** PCI-DSS v4.0 Requirements 3 (Protect Stored Data), 6 (Develop and Maintain Secure Systems), 7 (Restrict Access), 9 (Restrict Physical Access — AWS handles), 10 (Log and Monitor), 12 (Support Information Security).

**Non-Scope:** Requirements 1-2 (Network Controls — Connect is SaaS, no customer-managed network), Requirement 4 (Encryption in Transit — TLS enforced by Connect and all S3/API policies), Requirement 5 (Anti-Malware — AWS-managed).

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — Compliance Guard
All resources: `count = var.compliance_enabled ? 1 : 0`.

### FR-002 — Manual S3 Object Lock Procedure for Recordings Bucket
Document the manual S3 Object Lock procedure for the call recordings bucket (PRD-30). If an assessor requires governance-mode or compliance-mode immutability, operations must perform a change-controlled manual bucket recreation and then manually set the default Object Lock retention outside Terraform. Terraform and module code must not set `object_lock_enabled` or program the Object Lock mode or retention period. The approved change record must capture the selected lock mode, retention period, execution timestamp, and operator evidence.

### FR-003 — Dedicated KMS Key for Recordings
Provision a dedicated KMS CMK `{org_name}-recordings-pci-{environment}` used exclusively for call recordings encryption. This provides key isolation between recording data and other platform data. The key policy must preserve normal KMS administration while also allowing the Amazon Connect storage-association caller to create and retire the grants that Amazon Connect requires for the instance service-linked role. This module must not describe recordings encryption as a Lambda-only decrypt path, because call recordings are written through the Amazon Connect and S3 integration path rather than through a PRD-40 Lambda runtime.

### FR-004 — KMS Key Rotation Policy
Enforce automatic annual KMS key rotation on all platform CMKs. This was enabled in PRD-02 but is verified and re-enforced here as an explicit PCI control with documented evidence.

### FR-005 — Enhanced CloudTrail Configuration
Enable CloudTrail data events for S3 on the recordings and voicemail buckets, and for DynamoDB on the Contact State table. This may be implemented either as a dedicated compliance trail owned by PRD-140 or as an explicitly approved extension to an existing trail. The contract must name the chosen ownership model instead of assuming silent mutation of PRD-03. Use advanced event selectors when the target account already has selector-heavy trails and cost scoping matters.

### FR-006 — AWS Config PCI Rules
Provision AWS Config managed rules for PCI-DSS compliance:
- `s3-bucket-server-side-encryption-enabled` — recordings, voicemail, artifacts buckets
- `kms-cmk-not-scheduled-for-deletion` — all platform CMKs
- `cloudtrail-enabled` — organization-level trail
- `iam-no-inline-policy-check` — all platform IAM roles
- `access-keys-rotated` — service accounts

Non-compliance alerting must be delivered through the account's AWS Config SNS delivery channel or an explicit EventBridge rule on AWS Config compliance-change events. The config rule resources themselves do not own an SNS action.

### FR-007 — PCI Control Evidence Export
Maintain a current PCI control evidence manifest at `s3://{audit_bucket}/pci-evidence/current/manifest.json` enumerating the current state of each PCI control with its compliance status. Historical snapshots may be preserved by bucket versioning or by an explicit workflow export step, but the Terraform-owned object key must be deterministic so the module does not force perpetual drift.

### Manual Object Lock Change Procedure

**DESTRUCTIVE — Schedule a maintenance window and approved change record before executing.**

1. Confirm that the assessor or compliance owner explicitly requires Object Lock for the recordings bucket and record the chosen lock mode and retention period in the change request.
2. Export or copy the existing recordings to a temporary location because S3 Object Lock cannot be added in place to an existing bucket.
3. Recreate the recordings bucket manually with Object Lock capability enabled using an approved operator procedure outside Terraform. Do not add `object_lock_enabled` to PRD-30 or this module.
4. Manually apply the default Object Lock retention configuration in the chosen governance or compliance mode and record the exact values in the approved change evidence.
5. Restore the recordings data, re-establish the Connect storage association, and capture post-change verification that the bucket reports the intended Object Lock configuration.
6. Store the change ticket, operator evidence, and verification output in the audit bucket so the compliance package can point to the manual control evidence without implying Terraform ownership.

## 2. CONTEXT & PURPOSE

SOC 2 Type II requires a 6-12 month continuous audit period demonstrating that controls operate effectively over time. This PRD automates the evidence collection that auditors require: weekly control snapshots, change management records, access review reports, and incident response documentation. Evidence is collected continuously for the enabled modules and profiles and organized in the audit bucket by control category so that the auditor package can be assembled from S3 in minutes rather than weeks.

**Trust Service Criteria covered:** CC6 (Logical and Physical Access), CC7 (System Operations), CC8 (Change Management), A1 (Availability).

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — SOC 2 Control Inventory
Provision S3 object `s3://{audit_bucket}/soc2/control-inventory.json` enumerating all SOC 2 controls in scope and their implementation evidence pointers. Updated on every Terraform apply.

### FR-002 — Weekly Evidence Collection Lambda
Provision Lambda `{org_name}-soc2-evidence-collector-{environment}` running every Monday at 06:00 UTC (30 minutes before org-level evidence export). Collects and writes the following evidence to `s3://{audit_bucket}/soc2/evidence/{YYYY}/{WW}/`:

| Evidence Type | Source | File |
|---|---|---|
| IAM role inventory | `iam:ListRoles` | `iam-roles.json` |
| Active MFA on IAM users | `iam:ListVirtualMFADevices` | `mfa-status.json` |
| S3 bucket encryption configs | `s3:GetBucketEncryption` (all platform buckets) | `s3-encryption.json` |
| KMS key rotation status | `kms:GetKeyRotationStatus` | `kms-rotation.json` |
| CloudTrail status | `cloudtrail:GetTrailStatus` | `cloudtrail-status.json` |
| Config compliance summary | `config:GetComplianceSummaryByConfigRule` | `config-compliance.json` |
| Retention policy verification result (PRD-32) | S3 read of latest retention verification | `retention-verification.json` |
| Lambda function inventory with last-modified | `lambda:ListFunctions` | `lambda-inventory.json` |
| Security Hub findings summary | `securityhub:GetFindings` (HIGH/CRITICAL) | `security-hub-findings.json` |

### FR-003 — Change Management Log
Every Terraform apply for an in-scope environment must result in a change management record at `s3://{audit_bucket}/soc2/change-management/{YYYY}/{MM}/{DD}/{run_id}.json`. This is a PRD-01 workflow responsibility, not a Terraform resource owned by this module. The record includes: timestamp, applied modules, GitHub run ID, actor, environment, and the summarized resource-change count.

### FR-004 — Access Review Report Lambda
Provision Lambda `{org_name}-soc2-access-review-{environment}` running quarterly (1st Monday of each quarter at 07:00 UTC). Generates an access review report:
- All Connect users and their security profiles
- All IAM roles and their last-used dates
- Any IAM user access keys older than 90 days (should be none — platform uses OIDC)

Report written to `s3://{audit_bucket}/soc2/access-reviews/{YYYY}/{quarter}/report.json`.

### FR-005 — SOC 2 Auditor Package Generator
Provision Lambda `{org_name}-soc2-package-generator-{environment}` invocable on demand. Accepts `{ "period_start": "YYYY-MM-DD", "period_end": "YYYY-MM-DD" }`. It assembles all evidence from the audit period into a single ZIP archive at `s3://{audit_bucket}/soc2/auditor-packages/{period_start}_{period_end}.zip` and returns the object key. A pre-signed download URL is optional and may only be returned when an explicit operator-controlled flag allows temporary auditor access links.

## 2. CONTEXT & PURPOSE

Security Hub aggregates findings from AWS Config, Amazon Inspector, Amazon GuardDuty, and IAM Access Analyzer into a single compliance dashboard. This PRD enables the AWS Foundational Security Best Practices (FSBP) standard and the PCI DSS standard in Security Hub when the workload account is self-managed, or consumes delegated-admin central configuration as an explicit prerequisite when the organization uses centrally managed Security Hub. It configures automated remediation for a subset of findings using AWS Systems Manager Automation documents and provisions the Security Hub findings export to the SOC 2 evidence pipeline.

This PRD is the capstone of the optional compliance hardening profile. When enabled alongside the rest of the compliance layer, the platform gains continuous, automated compliance evidence collection covering both PCI-DSS and SOC 2 Type II from day one of the audit period.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — Security Hub Standards Activation
Enable the following Security Hub standards:
- AWS Foundational Security Best Practices v1.0.0
- PCI DSS v4.0.1

In `self_managed` mode, enable them via `aws_securityhub_standards_subscription` resources. In `central_config` mode, this module must not attempt to subscribe the account locally; instead it verifies that the delegated-admin configuration policy already enables the required standards.

### FR-002 — GuardDuty Enable
Provision `aws_guardduty_detector` in each workload account. GuardDuty findings are automatically aggregated into Security Hub. At minimum, enable S3 protection and Kubernetes protection (if EKS is used — set `var.enable_k8s_protection = false` for this platform).

### FR-003 — Automated Remediation for Critical Findings
Provision EventBridge rule on Security Hub CSPM findings matching severity `CRITICAL` or `HIGH`. Route to a remediation router Lambda `{org_name}-security-hub-remediator-{environment}` that keys off `SecurityControlId` rather than fragile title text or `GeneratorId` parsing:
- For `S3.1` (S3 bucket should not allow public read access): runs the `AWS-DisableS3BucketPublicReadWrite` SSM automation document with its documented `S3BucketName` parameter
- For `KMS.4` (KMS rotation should be enabled): enables rotation via `kms:EnableKeyRotation`
- For `CloudTrail.1` (CloudTrail should be enabled): alerts operations team — remediation cannot be automated without risk
- For all others: publishes to platform SNS alert topic for manual review

### FR-004 — Security Hub Findings Export to SOC 2 Pipeline
The evidence collector Lambda from PRD-141 reads Security Hub findings. This PRD supplements that by provisioning a daily EventBridge rule that exports all CRITICAL and HIGH findings to `s3://{audit_bucket}/soc2/security-hub/daily/{date}/findings.json`. Provides continuous evidence that findings are reviewed and remediated within the required SLA.

### FR-005 — Finding SLA Tracking
Provision DynamoDB table `{org_name}-finding-sla-{environment}` tracking each Security Hub finding's open/closed status and time to remediation. Used by the SOC 2 package generator (PRD-141) to demonstrate mean time to remediation (MTTR) metrics for auditors.

### Lambda Source (Remediation Logic)

```python
# lambda-src/security-hub-remediator/index.py
import os
import json
import boto3
import logging
from datetime import datetime, timezone, timedelta

s3       = boto3.client('s3')
ssm      = boto3.client('ssm')
kms      = boto3.client('kms')
sechub   = boto3.client('securityhub')
sns      = boto3.client('sns')
dynamodb = boto3.resource('dynamodb')

AUDIT_BUCKET      = os.environ['AUDIT_BUCKET']
FINDING_SLA_TABLE = os.environ['FINDING_SLA_TABLE']
ALERT_TOPIC_ARN   = os.environ['PLATFORM_ALERT_TOPIC_ARN']
table = dynamodb.Table(FINDING_SLA_TABLE)
LOGGER = logging.getLogger()
LOGGER.setLevel(logging.INFO)

AUTOMATED_REMEDIATIONS = {
    'S3.1':   lambda f: ssm.start_automation_execution(
                DocumentName='AWS-DisableS3BucketPublicReadWrite',
                Parameters={'S3BucketName': [f['Resources'][0]['Id'].split(':')[-1]]}),
    'KMS.4':  lambda f: kms.enable_key_rotation(KeyId=f['Resources'][0]['Id'].split('/')[-1]),
}

def handler(event, context):
    action = event.get('action', 'remediate')

    if action == 'export_daily_findings':
        export_daily_findings()
        return {'statusCode': 200}

    for finding in event.get('detail', {}).get('findings', []):
        finding_id   = finding.get('Id', '')
        control_id   = finding.get('SecurityControlId', '')
        severity     = finding.get('Severity', {}).get('Label', 'UNKNOWN')
        title        = finding.get('Title', '')

        # Record in SLA table
        table.put_item(Item={
            'FindingId':      finding_id,
            'SecurityControlId': control_id,
            'Severity':       severity,
            'Title':          title,
            'Status':         'OPEN',
            'OpenedAt':       datetime.now(timezone.utc).isoformat(),
            'ExpiresAt':      int((datetime.now(timezone.utc) + timedelta(days=365)).timestamp())
        })

        # Attempt automated remediation
        if control_id in AUTOMATED_REMEDIATIONS:
            try:
                AUTOMATED_REMEDIATIONS[control_id](finding)
                table.update_item(
                    Key={'FindingId': finding_id},
                    UpdateExpression='SET #s = :s, RemediatedAt = :ra',
                    ExpressionAttributeNames={'#s': 'Status'},
                    ExpressionAttributeValues={':s': 'AUTO_REMEDIATED', ':ra': datetime.now(timezone.utc).isoformat()}
                )
                LOGGER.info("Auto-remediated finding %s %s", control_id, finding_id)
            except Exception as e:
                LOGGER.error("Auto-remediation failed %s %s", control_id, str(e))
                alert_operations(finding, f"Auto-remediation failed: {e}")
        else:
            alert_operations(finding, f"Manual remediation required for {control_id or 'UNKNOWN_CONTROL'}: {title}")

    return {'statusCode': 200}

def alert_operations(finding, message):
    sns.publish(
        TopicArn=ALERT_TOPIC_ARN,
        Subject=f"Security Hub Finding — {finding.get('Severity', {}).get('Label', 'UNKNOWN')}: {finding.get('Title', '')}",
        Message=json.dumps({'message': message, 'finding': finding}, indent=2)
    )

def export_daily_findings():
    findings = []
    paginator = sechub.get_paginator('get_findings')
    for page in paginator.paginate(
        Filters={
            'SeverityLabel':  [{'Value': 'CRITICAL', 'Comparison': 'EQUALS'}, {'Value': 'HIGH', 'Comparison': 'EQUALS'}],
            'RecordState':    [{'Value': 'ACTIVE', 'Comparison': 'EQUALS'}]
        },
        PaginationConfig={'PageSize': 100}
    ):
        findings.extend(page.get('Findings', []))
    date       = datetime.now(timezone.utc).strftime('%Y/%m/%d')
    s3_key     = f"soc2/security-hub/daily/{date}/findings.json"

    s3.put_object(
        Bucket=AUDIT_BUCKET, Key=s3_key,
        Body=json.dumps({'exported_at': datetime.now(timezone.utc).isoformat(), 'count': len(findings), 'findings': findings}),
        ContentType='application/json'
    )
    LOGGER.info("Daily Security Hub findings exported %s %s", len(findings), s3_key)
```
