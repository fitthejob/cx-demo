## 3. CONTEXT & PURPOSE

The CRM adapter layer provides a CRM-agnostic interface between the platform and any CRM system. Real-time lookups may be invoked directly from Connect and adjacent workflows, while asynchronous updates may be delivered through the event-driven profile when PRD-20 is enabled. This PRD owns the canonical CRM entry-point Lambda functions used by the rest of the platform. Provider-specific PRDs such as PRD-131 supply credentials, provider configuration, and the deployment artifact that PRD-41 publishes to those canonical functions. The provider switch is therefore a controlled deployment input, not a cross-module resource takeover.

**CRM Toggle:** When `crm_enabled = false` (default), PRD-71's `CheckAccountStatus` intent returns "our agents will assist you" per the design established in PRD-71/72. When `crm_enabled = true`, the CheckAccountStatus flow branch invokes the CRM lookup Lambda defined by PRD-130 and implemented by the active provider module such as PRD-131.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — CRM Guard
All resources are conditional on the manifest-selected CRM pack and the runtime toggle: `count = local.crm_enabled ? 1 : 0`.

### FR-002 — CRM Adapter Interface
The CRM adapter must expose two canonical Lambda functions with a fixed interface contract regardless of which CRM is behind it:
- `{org_name}-crm-lookup-{environment}` — accepts a contact ID and caller E.164 number; returns CRM contact record
- `{org_name}-crm-update-{environment}` — accepts a contact ID and update payload; updates the CRM record

These function names are fixed. PRD-131 provides the HubSpot provider package and secret contract. A future Salesforce migration may change the deployed artifact and provider credentials, but it must not change the function names or the request/response contract consumed by downstream layers.

### FR-003 — CRM Lookup Response Schema
The CRM lookup Lambda must return a fixed JSON schema regardless of the underlying CRM:

```json
{
  "found": true,
  "contact_id": "CRM-system-record-id",
  "contact_name": "string",
  "contact_email": "string",
  "record_url": "string-or-empty",
  "account_name": "string",
  "account_status": "string",
  "recent_interactions": [
    { "date": "ISO 8601", "type": "call|email|ticket", "summary": "string" }
  ],
  "custom_fields": {}
}
```

### FR-004 — CRM Event Subscriptions
When PRD-20 is enabled, provision EventBridge rules for the following base events, routing to the CRM update Lambda:
- `ConnectPBX.ContactCompleted` → append call outcome or summary data to an existing CRM record when a CRM contact ID is available
- `ConnectPBX.VoicemailTranscribed` → attach transcription or voicemail summary data to an existing CRM record when a CRM contact ID is available

`ConnectPBX.AgentProvisioned` is not part of the base Layer 13 implementation-ready contract. If a future CRM provider needs agent-record synchronization, that path must be added as an explicit provider-specific extension with its own event schema and rollout notes.

### FR-005 — CRM Provider Rollout Contract
`var.crm_provider` accepts `hubspot` or `salesforce` (default `hubspot`). The selected provider determines which provider package, secret material, and configuration values PRD-41 publishes to the canonical PRD-130 Lambda functions. Terraform in PRD-130 does not directly replace another module's Lambda resources, and provider rollouts must be sequenced as deployment changes to the canonical functions rather than hidden cross-module edits.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — HubSpot API Credentials
Store HubSpot private app access token in AWS Secrets Manager: `{org_name}/crm/hubspot/access-token`. The Lambda retrieves this secret at runtime — never hardcoded in environment variables.

### FR-002 — HubSpot Contact Lookup
The `crm-lookup` Lambda implementation calls HubSpot's contact search endpoint with an HTTP `POST` request and the caller's phone number in the request body filter set. It returns the first matching contact mapped to the CRM adapter response schema from PRD-130. The implementation must account for HubSpot's documented phone-search normalization behavior and must not assume the raw E.164 value will match unchanged.

### FR-003 — HubSpot Contact Update
The `crm-update` Lambda implementation creates HubSpot call and note activities with HTTP `POST` requests to the documented calls and notes object endpoints. When a `contact_id` is present, it must associate the new call or note with that existing contact record. If the event does not contain a CRM contact ID, the Lambda must log a structured warning and return success without pretending that a timeline association was created.

### FR-004 — HubSpot Rate Limit Handling
The Lambda must implement exponential backoff with jitter and must treat HubSpot's current documented limits as configuration-sensitive rather than hardcoded. This matters because privately distributed apps have account-tier-dependent burst limits, while the CRM Search API also has its own stricter documented rate limit. Rate limit errors (HTTP 429) must raise `TransientError` for Lambda retry. The provider package may use shared retry helpers when PRD-40 is enabled, but it must also support a self-contained retry implementation.

### FR-005 — Optional Record URL Construction
When `hubspot_portal_id` is supplied, the lookup Lambda may construct a deep link for the matched contact and return it in `record_url`. When that input is absent, the Lambda must return an empty string rather than inventing a provider URL format. PRD-133 uses the optional deep link when present and falls back cleanly when it is not.

```python
# HubSpot crm-lookup implementation
import os
import time
import random
import boto3
import requests
import logging

secrets_client = boto3.client('secretsmanager')
LOGGER = logging.getLogger()
LOGGER.setLevel(logging.INFO)

class TransientError(Exception):
    pass

class PermanentError(Exception):
    pass

def get_token():
    secret = secrets_client.get_secret_value(SecretId=os.environ["HUBSPOT_SECRET_NAME"])
    return secret['SecretString']

def retry_request(method, url, *, headers, json_body):
    for attempt in range(1, 4):
        response = requests.request(method, url, headers=headers, json=json_body, timeout=5)
        if response.status_code != 429:
            return response
        if attempt == 3:
            raise TransientError("HubSpot rate limit exceeded")
        time.sleep(min(2 ** attempt, 8) + random.uniform(0, 0.5))
    raise TransientError("HubSpot rate limit exceeded")

def hubspot_search(phone_number, token):
    resp = retry_request(
        "POST",
        os.environ["HUBSPOT_CONTACT_SEARCH_URL"],
        headers={"Authorization": f"Bearer {token}", "Content-Type": "application/json"},
        json_body={
            "filterGroups": [{
                "filters": [
                    {"propertyName": "phone", "operator": "EQ", "value": phone_number},
                    {"propertyName": "mobilephone", "operator": "EQ", "value": phone_number}
                ]
            }],
            "properties": ["firstname", "lastname", "email", "company", "hs_object_id"]
        }
    )
    if resp.status_code not in (200, 201):
        raise PermanentError(f"HubSpot API error: {resp.status_code} {resp.text}")
    return resp.json()

def record_url(contact_id):
    portal_id = os.environ.get("HUBSPOT_PORTAL_ID", "").strip()
    if not portal_id:
        return ""
    return f"https://app.hubspot.com/contacts/{portal_id}/record/0-1/{contact_id}"

def handler(event, context):
    payload     = event.get('Details', {}).get('Parameters', event)
    caller_num  = payload.get('caller_number') or event.get('Details', {}).get('ContactData', {}).get('CustomerEndpoint', {}).get('Address', '')

    if not caller_num:
        return {"found": False, "record_url": ""}

    token   = get_token()
    result  = hubspot_search(caller_num, token)
    results = result.get('results', [])

    if not results:
        return {"found": False, "record_url": ""}

    contact    = results[0]
    props      = contact.get('properties', {})
    first_name = props.get('firstname', '')
    last_name  = props.get('lastname', '')
    contact_id = contact.get('id', '')

    return {
        "found":          True,
        "contact_id":     contact_id,
        "contact_name":   f"{first_name} {last_name}".strip(),
        "contact_email":  props.get('email', ''),
        "record_url":     record_url(contact_id),
        "account_name":   props.get('company', ''),
        "account_status": "",
        "recent_interactions": [],
        "custom_fields": {}
    }
```

## 3. CONTEXT & PURPOSE

When the CRM lookup Lambda (PRD-130 plus an enabled provider implementation such as PRD-131) returns contact data, that data must be mapped back to Connect contact attributes so downstream services (whisper flows PRD-52, screen pop PRD-133) can access it. This PRD provisions the attribute mapping Lambda and defines two distinct operating modes:

- the canonical path is synchronous invocation from the Connect flow before whisper logic runs
- the optional EventBridge path is best-effort enrichment or repair and must not be treated as the whisper-critical contract

---

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Attribute Mapping Lambda
Provision Lambda `{org_name}-contact-attribute-mapper-{environment}`. In lean profiles it is invoked directly from the Connect interaction path. When PRD-20 is enabled it may also be triggered by an EventBridge rule on `ConnectPBX.ContactInitiated` for best-effort enrichment or repair only. The Lambda:
1. Extracts caller phone number from event payload
2. Invokes `{org_name}-crm-lookup-{environment}:LIVE`
3. Maps CRM response fields into a flat string map with these keys:
   - `crm_found` → `"true"` or `"false"`
   - `CRMContactId` → CRM contact record ID or empty string
   - `CRMContactName` → full contact name (used by PRD-52 whisper flow)
   - `CRMRecordUrl` → CRM record URL or empty string (used by PRD-133 screen pop)
   - `CRMAccountName` → account/company name or empty string
   - `CRMAccountStatus` → account status or empty string
4. On the synchronous Connect path, returns that flat string map to the flow. The consuming flow must explicitly copy the returned values into contact attributes using `Set contact attributes` before the whisper branch executes.
5. On the optional EventBridge path, may call Connect `UpdateContactAttributes` only when the event includes the required Connect identifiers and the operator accepts that the update is eventual rather than whisper-critical
6. Updates the Contact State DynamoDB table with `CRMContactId` and `CRMRecordUrl` only when PRD-31 shared state is enabled

### FR-002 — Latency Requirement
The synchronous pre-whisper path must complete within 3 seconds to avoid affecting the caller's perceived wait time. This is the only path that can be used to satisfy the agent-whisper requirement in PRD-52. The optional EventBridge path is not a substitute for that timing-sensitive contract.

## 3. CONTEXT & PURPOSE

Screen pop delivers the CRM record URL to the agent's browser the moment a call connects. The agent sees the caller's CRM profile automatically without manual searching. This PRD implements screen pop using the Connect Streams API (JavaScript) and exports the configuration parameters for embedding in agent workspace applications.

Screen pop is implemented as a JavaScript snippet that subscribes to the Connect Streams API `contact.onConnected` event and opens the CRM record URL from the `CRMRecordUrl` contact attribute. It is not a Terraform resource by itself — it is browser code that must be loaded by an approved PRD-51 workspace origin. This PRD may deliver the script/config as rendered S3 objects, or it may hand those rendered assets to an operator-managed web application pipeline. It must not assume a public S3 URL, a central PRD-40 bucket, or a hidden browser-hosting prerequisite.

---

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Screen Pop Asset Delivery Modes
Support two explicit delivery modes:
- `s3_objects` — render `config.json` and `screen-pop.js` into either a module-owned bucket or an explicitly supplied shared bucket
- `external_workspace_bundle` — render the same assets for handoff to an operator-managed web application pipeline without creating S3 objects

In both modes, the module must produce the same runtime configuration contract.

### FR-002 — Screen Pop Configuration Contract
Render a configuration payload containing:
```json
{
  "enabled": true,
  "crm_url_attribute": "CRMRecordUrl",
  "open_on": "connected",
  "fallback_url": ""
}
```

### FR-003 — Streams API Screen Pop Script
Render a Connect Streams API script that the agent workspace application loads. The script must read its config from an explicit asset URL or in-page bootstrapped config value, not from an inferred public S3 website path.

### FR-004 — CCP Embedding Integration
Export the rendered asset location needed by the agent workspace application. The workspace team remains responsible for loading the script alongside the Connect CCP embed from PRD-51 and for ensuring the workspace origin is on the Amazon Connect approved-origins list.

    // Loaded by agent workspace application alongside CCP embed (PRD-51)

    (function() {
      'use strict';

      const CONFIG_URL = window.CONNECT_PBX_SCREEN_POP_CONFIG_URL;

      async function loadConfig() {
        if (!CONFIG_URL) {
          console.warn('[ScreenPop] No config URL provided');
          return { enabled: false };
        }
        const resp = await fetch(CONFIG_URL);
        return await resp.json();
      }

      async function initScreenPop() {
        const config = await loadConfig();
        if (!config.enabled) return;

        connect.contact(function(contact) {
          contact.onConnected(function() {
            const attributes = contact.getAttributes();
            const crmUrl     = attributes[config.crm_url_attribute]?.value;
            const crmName    = attributes[config.crm_name_attribute]?.value;

            if (crmUrl) {
              window.open(crmUrl, 'crm-screen-pop', 'width=1200,height=800,noopener');
              console.log('[ScreenPop] Opened CRM record for:', crmName || 'Unknown');
            } else if (config.fallback_url) {
              window.open(config.fallback_url, 'crm-screen-pop', 'width=1200,height=800,noopener');
            }
          });
        });
      }

      if (typeof connect !== 'undefined') {
        initScreenPop();
      } else {
        document.addEventListener('DOMContentLoaded', initScreenPop);
      }
    })();
  JS
