## 3. CONTEXT & PURPOSE

Provisions CloudWatch alarms for enabled platform-critical metrics and optionally routes alarm notifications to shared SNS, email, and PagerDuty endpoints when those sinks are configured. Alarm resources should be created only for modules that are actually enabled in the environment.

---

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Alarm Consolidation
Provision CloudWatch metric alarms for a curated base set plus explicit optional alarm families selected through module inputs. This module must not blindly mirror every historical `ALARM-XX-YY` annotation into Terraform. Each alarm attaches `alarm_action_arns` only when those optional action ARNs are provided.

Required base alarm set:

| Alarm ID | Metric | Threshold | Severity |
|---|---|---|---|
| ALARM-10-01 | CallsBreachingConcurrencyQuota | > 0 | Critical |
| ALARM-10-02 | ContactFlowFatalErrors | > 0 in 5min | High |
| ALARM-13-01 | QueueSize per queue | > 20 | High |
| ALARM-13-03 | AgentsAvailable = 0 during hours | 0 | Critical |

Optional alarm families may be added only when the corresponding source modules are enabled and their metric contracts are wired into this module via explicit inputs such as queue maps, Lambda names, DLQ names, or custom metric names. Examples include EventBridge DLQ depth, voicemail processing alarms, Lex alarms, and later telecom-governance modules.

### FR-002 — SNS Subscriptions
When `platform_alert_topic_arn` is provided, provision SNS subscriptions on that topic:
- **dev/staging:** Email subscription to `var.alert_email`
- **prod:** Email subscription + optional HTTPS endpoint subscription for PagerDuty integration (`var.pagerduty_webhook_url`)

### FR-003 — Composite Alarm
Provision a composite alarm `ConnectPBX-Platform-Critical-{environment}` that fires when any single enabled Critical-severity alarm is in ALARM state. This is the top-level alarm for executive-level dashboards.

## 3. CONTEXT & PURPOSE

Without cost allocation tags on phone numbers, Lambda functions, and S3 buckets, costs are invisible by department. ## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Cost Allocation Tags
Document and validate activation of the following tag keys as cost allocation tags in AWS Billing:
- `Project` — all resources
- `Layer` — architecture layer
- `CostCenter` — queue/department allocation
- `Environment` — dev/staging/prod split

This activation is management-account scoped and sits outside the standard workload module deployment path. The workload module may validate that required tags are present and surface operator guidance, but must not require a management-account apply as part of routine environment deployment.

### FR-002 — Cost Anomaly Detection
Provision `aws_ce_anomaly_monitor` for the Connect service and, when `cost_alert_topic_arn` is provided, provision `aws_ce_anomaly_subscription` that alerts through that optional topic when costs exceed `var.anomaly_threshold_usd` (default $50). SNS-backed anomaly subscriptions must use `frequency = "IMMEDIATE"`. Daily or weekly summaries, if desired, are email-only follow-on reporting paths rather than SNS alarm transports.

### FR-003 — FinOps Dashboard
Provision `aws_cloudwatch_dashboard` named `ConnectPBX-FinOps-{environment}` as an operator landing page, not as a direct Cost Explorer graph surface. The dashboard may contain:
- Text widgets linking operators to Cost Explorer and Billing views
- Optional custom CloudWatch metrics published by the weekly cost reporter when `enable_finops_estimate_metrics = true`
- Native workload metrics that help explain cost movement, such as Lambda invocations or S3 storage trends

The module must not claim that CloudWatch dashboards can render native Cost Explorer widgets.

### FR-004 — Weekly Cost Report Lambda
Provision Lambda `{org_name}-cost-reporter-{environment}` on a Monday 08:00 UTC schedule when `enable_weekly_cost_report = true`. The Lambda queries AWS Cost Explorer, breaks down costs by `CostCenter` tag for the previous 7 days, writes the report to `s3://{cost_report_bucket}/cost-reports/{YYYY}/{MM}/{DD}/weekly-report.json` when a report bucket sink is configured, optionally sends an email summary via SES if `var.cost_report_email` is set, and may publish custom CloudWatch estimate metrics only when `enable_finops_estimate_metrics = true`.
