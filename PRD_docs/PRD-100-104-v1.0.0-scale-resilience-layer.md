## 2. MODULE GOVERNANCE

### Shared Sink Behavior

- `optional_shared_sinks`: none

### Destroy / Retention Posture

- `destroy_posture`: `destroyable`
- `retention_notes`: PRD-100 should own only additive HA tuning resources. It must not take ownership of retained phone numbers, Connect instances, or other lower-layer stateful data.

### Control Plane Statement

PRD-100 may coordinate runtime-shape behavior for enabled lower-layer modules, but it must do so only through explicit Layer 10-owned additive resources and declared inputs. It must not become a hidden prerequisite for baseline telephony, and it must not silently mutate queue, table, or Lambda configuration that is owned by another module. If implementation requires reading lower-layer remote state, those reads must be mirrored as explicit dependencies before the PRD can move to implementation.

## 3. CONTEXT & PURPOSE

Amazon Connect is inherently multi-AZ. The surrounding infrastructure is not. PRD-100 defines the enterprise runtime-shape contract for multi-AZ promotion across the modules that are already enabled in an environment.

Activation is two-step:

1. enable the Layer 10 module through the module catalog and environment deployment manifest
2. set the runtime-shape inputs that require HA behavior, such as `deployment_profile.multi_az = true`

This PRD is not required for bare-bones or small deployments.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Runtime-Shape Guard

When `deployment_profile.multi_az = true`, PRD-100 may create only explicit Layer 10-owned additive resources for targets that are deliberately supplied to the module. Setting the flag alone must not enable PRD-100 in an environment, and it must not retroactively rewrite lower-layer module configuration.

Allowed additive examples in v1:

| Enabled Surface | Multi-AZ Runtime-Shape Impact |
|---|---|
| PRD-40 Lambda aliases, if explicitly supplied | Provisioned concurrency on selected hot-path aliases |
| CTR/Kinesis ingestion path, if explicitly supplied | Enhanced fan-out consumer registration or equivalent stream-side additive scaling resource |
| Layer 10 runtime consumers | SSM parameters or module outputs that publish the selected HA posture to downstream operators and automation |

### FR-002 — Additive HA Controls Only

PRD-100 may provision additive runtime-shape resources such as provisioned concurrency or consumer registrations, but it must not edit another module's source files, take ownership of lower-layer resources, or rely on manual `terraform state rm` as a steady-state boundary.

### FR-003 — Optional Eventing And Stream Inputs

If an environment does not enable Kinesis or the event-driven reliability path, PRD-100 must not assume those resources exist. HA tuning for those surfaces must be modeled as optional inputs or omitted entirely.

### FR-004 — No Hidden Universal Prerequisites

PRD-100 must not make PRD-21, PRD-31, or PRD-40 universal requirements for all Layer 10 deployments. Any integration with those modules must be explicit and conditional on them being enabled in the manifest.

## 6. ACCEPTANCE CRITERIA

- Applying PRD-100 with `multi_az = false` creates no additive HA resources.
- Applying PRD-100 with `multi_az = true` only creates additive Layer 10-owned resources for targets that are both enabled and explicitly wired into the runtime-shape contract.
- The PRD does not mutate lower-layer queue, table, or function configuration in place unless the affected resource is explicitly modeled as a Layer 10 additive resource.
- The PRD does not require PRD-21, PRD-31, PRD-40, Kinesis, or PRD-03 unless those integrations are intentionally enabled and declared.

## 3. CONTEXT & PURPOSE

When scale, isolation, or departmental segmentation requires more than one Connect instance, PRD-101 defines the additive routing registry and topology contracts needed to support that model. It is not part of the default single-instance baseline.

Activation is two-step:

1. enable the Layer 10 multi-instance module through the manifest/catalog control plane
2. set `deployment_profile.instance_count > 1` and provide the routing inputs needed for the selected topology

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Runtime Topology Guard

All PRD-101 resources must be conditional on an enabled Layer 10 module and an intentional multi-instance runtime shape. `instance_count = 1` remains the default no-op posture.

### FR-002 — Department-To-Instance Routing Registry

When multi-instance mode is selected, v1 must provision only an additive routing registry surface, not a magical global ingress layer. The concrete v1 contract is:

- a routing table keyed by department or service domain
- optional operator-facing outputs or parameters that publish the primary and secondary instance assignments
- optional router Lambda only when an explicit entry-point strategy is separately supplied

The PRD must not imply that Amazon Connect natively performs department-based routing across instances without an explicit ingress strategy.

### FR-002A — Explicit Ingress Strategy Prerequisite

If live inbound distribution across instances is required, the environment must declare an explicit ingress strategy such as:

- dedicated phone-number ownership per instance
- a carrier, SBC, or upstream telephony layer that chooses the destination instance
- an operator-controlled entry-number pattern

PRD-101 must not claim that a single default Connect phone number can be routed across multiple independent instances without such a declared strategy.

### FR-003 — Secondary Instance Ownership Boundary

Secondary Connect instance lifecycle is out of default v1 scope unless separately managed as explicit additional PRD-10 deployments. PRD-101 must not edit PRD-10 source files or assume that every Layer 1 through Layer 8 module becomes multi-instance by default.

### FR-004 — Lower-Layer Consumer Implications Are Opt-In

PRD-15 through PRD-19, and other service modules, may later accept multi-instance-aware inputs when enabled. Those are future optional integrations, not present-tense universal dependencies that PRD-101 can force into lower layers.

- any remote-state reads from lower-layer modules must be promoted to explicit catalog dependencies before implementation

## 6. ACCEPTANCE CRITERIA

- With `instance_count = 1`, PRD-101 creates no multi-instance resources.
- With `instance_count > 1`, PRD-101 provisions only the additive routing-registry and topology artifacts explicitly defined for the rollout.
- The PRD does not claim automatic secondary-instance provisioning or automatic cross-instance inbound routing without an explicit ingress strategy.

## 3. CONTEXT & PURPOSE

PRD-102 defines the additive regional continuity posture for environments that intentionally adopt cross-region failover. For Amazon Connect telephony, the canonical v1 path is traffic distribution groups and `UpdateTrafficDistribution`, not generic DNS failover. The module may include secondary-region Connect capacity, replicated data planes, and optional non-telephony domain failover controls, but only for the modules that are both enabled and explicitly designed for that posture.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Cross-Region Runtime Guard

PRD-102 resources must remain inactive unless:

- the Layer 10 module is enabled through the manifest/catalog control plane
- the environment intentionally sets `deployment_profile.cross_region = true`
- the relevant lower-layer modules expose the inputs or contracts needed for cross-region rollout

### FR-002 — Optional Stateful Replication

Shared-state replication for PRD-30, PRD-31, or similar modules must remain conditional. PRD-102 must not imply that every stateful or optional layer becomes cross-region automatically.

### FR-003 — Additive Regional Routing

If Amazon Connect telephony failover is in scope, the module must use an explicit traffic distribution group contract:

- claimed numbers must already be assigned to the traffic distribution group
- telephony distribution must be updated through `UpdateTrafficDistribution`
- agent distribution and sign-in distribution must be shifted in lockstep with telephony

If Route 53 health checks, custom domains, or similar routing are used, those should be modeled as explicit inputs for non-telephony surfaces. The PRD must not assume `.my.connect.aws` defaults or generic PSTN DNS failover can redirect claimed Connect numbers without the required traffic-distribution strategy.

### FR-004 — No Universal Cross-Region Requirement

Cross-region continuity is an enterprise posture. PRD-102 must not backflow those requirements into baseline telephony, migration-only modules, or unrelated optional layers.

### FR-005 — Test Isolation Guard

Load-test execution must require explicit target declarations and a test-scope safety posture such as:

- approved non-production numbers, queues, or flows
- a maintenance window flag for production-adjacent validation
- test-scoped stop conditions and concurrency ceilings

The PRD must not assume that enterprise capacity testing can safely target all live caller surfaces by default.

### Control Plane Statement

## 3. CONTEXT & PURPOSE

Chaos engineering is an advanced operator capability used to validate resilience assumptions before real incidents. It is not part of the minimum platform deploy path and must not be treated as a prerequisite for lower layers.

## 4. FUNCTIONAL REQUIREMENTS

### FR-001 — Enabled-Capability Experiment Scope

Experiment templates must target only the resources that actually exist in the environment and are supported by the chosen fault-injection mechanism. The PRD must not claim universal EventBridge, Kinesis, voicemail, or Amazon Connect experiments when those capabilities are not enabled or when the underlying service is not a supported target type.

### FR-002 — Optional Game-Day Artifacts

Game-day runbooks and experiment evidence should be stored in a module-owned or explicitly supplied artifact sink. Hardcoding a shared audit bucket is not acceptable.

### FR-003 — Optional Dashboard And Alarm Wiring

Chaos metrics dashboards or stop-condition alarms may integrate with observability layers when present, but those integrations must remain optional and explicit.

### FR-004 — Experiment Safety Guard

All experiments require explicit stop conditions and operator safeguards to prevent spillover into live caller traffic.

### FR-005 — FIS Support Boundary

In v1, AWS Fault Injection Service experiments must be limited to AWS resource types that FIS actually supports, such as selected Lambda or Kinesis targets when those capabilities exist in the environment. Amazon Connect-managed instance, phone-number, and flow resources are not assumed to be direct FIS targets in this PRD. Connect-specific failover drills should therefore remain runbook-driven game days unless and until AWS exposes a supported direct experiment path.
