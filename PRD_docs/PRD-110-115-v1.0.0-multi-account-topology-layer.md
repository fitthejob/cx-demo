# PRD-110 — AWS Organizations & Management Account Setup

---

## Module Governance

This consolidated layer follows the current repo control plane:
- Module catalog plus per-environment manifest decide which capabilities are active.
- Layer 11 remains part of the target `enterprise-topology` pack family even while current manifests still use compatibility-era pack IDs.
- `deployment_profile` only carries runtime shape such as topology, counts, region, and capacity.
- Conditional foundations are allowed only when their upstream outputs are explicit inputs and not lower-layer backflow.
- Backend examples use logical `state_key` values; workspace scoping is supplied by the repo bootstrap convention.
- Optional features must not read lower-layer remote state unless the dependency is explicit and already enabled.

---

| Field | Value |

| **Blocks** | PRD-111 through PRD-115 |
| **Conditional foundation** | Yes — enterprise-profile foundation when `account_topology != "standalone"` and the manifest selects centralized audit delivery |

---

## 2. CONTEXT & PURPOSE

Activates AWS Organizations and creates the organizational unit (OU) structure for the platform. Activation is governed by the module catalog and deployment manifest; `deployment_profile` only selects runtime shape. This module activates the SCPs defined in PRD-02 (which were provisioned but not attached in standalone mode). When centralized audit delivery is enabled by the manifest, the Organization CloudTrail target is supplied as an explicit input from the manifest-selected shared-services audit sink. This is a management-account governance foundation for enterprise and multi-account profiles, not part of the default workload deployment path.

**Prerequisite:** This module is applied from the management account, not a workload account. The Terraform provider must be configured with management account credentials.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — AWS Organizations Structure
Provision the following organizational unit hierarchy:

```
Root
└── ConnectPBX
    ├── Workloads
    │   ├── Dev
    │   ├── Staging
    │   └── Prod
    └── Shared-Services
```

### FR-002 — SCP Attachment
Attach the SCPs defined in PRD-02 to the `ConnectPBX` OU:
- `deny-non-approved-regions` → attached to ConnectPBX OU
- `deny-cloudtrail-disable` → attached to ConnectPBX OU
- `deny-kms-key-deletion` → attached to ConnectPBX OU
- `deny-root-usage` → attached to ConnectPBX OU

### FR-003 — Organization CloudTrail
Provision an Organization-level CloudTrail trail that collects management events from all accounts in the organization. Deliver logs to the shared-services audit bucket passed in as an explicit input when centralized audit delivery is enabled.

### FR-004 — Enable AWS Services for Organization
Enable the following AWS services for use across the organization:
- `cloudtrail.amazonaws.com`
- `config.amazonaws.com`
- `securityhub.amazonaws.com`
- `access-analyzer.amazonaws.com`

# PRD-111 — Shared Services Account Architecture

---

## 1. METADATA

| Field | Value |
|---|---|
| **PRD ID** | PRD-111 |
| **Version** | 1.2.0 |
| **Layer** | 11 — Multi-Account Topology |
| **Depends On** | PRD-110 (Organizations) |
| **Conditional foundation** | Yes — enterprise-profile foundation for shared-services topologies selected by the manifest |

---

## 2. CONTEXT & PURPOSE

The Shared Services account is the hub for enterprise and multi-account deployments. It can host centralized observability, centralized audit collection, the optional central EventBridge bus, and optional delegated-admin service surfaces that are designated from the management account. The centralized EventBridge bus is only required when the event-driven capability pack is enabled in the manifest. IAM Identity Center remains an organization instance in the management account even when operations are delegated elsewhere, so this PRD must not claim to create the organization instance inside Shared Services. This PRD provisions the Shared Services account infrastructure following the same module patterns as the workload accounts but configured as the hub in the `account_topology = "hub"` runtime profile.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — Centralized EventBridge Bus
When the manifest enables the event-driven capability pack, provision a centralized EventBridge bus in the Shared Services account. This bus receives events from spoke account buses via cross-account event rules. Consumers in the Shared Services account subscribe to the centralized bus for cross-account analytics, CRM integration, and observability aggregation.

### FR-002 — Centralized Audit S3 Bucket
Provision an audit S3 bucket in the Shared Services account with bucket policies allowing all workload accounts to write to their respective prefixes. This centralizes evidence collection for SOC 2 and PCI audits.

### FR-003 — Centralized Security Hub
Enable Security Hub in the Shared Services account and expose its account ID as the delegated-admin target. The actual delegated administrator designation is performed from the management account as part of the PRD-110 organizational setup runbook or a management-account provider alias.

### FR-004 — AWS SSO (IAM Identity Center) Foundation
When the manifest enables the optional SSO layer, record the Shared Services account as the delegated-admin target for IAM Identity Center integrations if the organization instance has already been enabled in the management account. This PRD does not claim to create the IAM Identity Center organization instance in Shared Services.

---

# PRD-112 — Cross-Account EventBridge Mesh

---

## 1. METADATA

| Field | Value |
|---|---|
| **PRD ID** | PRD-112 |
| **Version** | 1.2.0 |
| **Layer** | 11 — Multi-Account Topology |
| **Depends On** | PRD-20 (workload event bus and schema registry, when the event-driven pack is enabled), PRD-111 (central event bus ARN) |
| **Conditional foundation** | Yes — conditional foundation for multi-account event-driven profiles selected by the manifest |

---

## 2. CONTEXT & PURPOSE

Creates EventBridge rules in each spoke account that forward `connect-pbx.*` events to the central bus in the Shared Services account. This PRD is only required when the manifest selects both the multi-account profile and the event-driven capability pack. It is not required for multi-account deployments that keep services local and direct. `deployment_profile.account_topology` only selects runtime shape; it does not activate this module.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — Cross-Account Forwarding Rule
In each spoke workload account, provision an EventBridge rule on the local custom bus that forwards all `connect-pbx.*` events to the central bus ARN in the Shared Services account.

### FR-002 — Cross-Account Target IAM
Provision an IAM role in each spoke workload account for the EventBridge rule target to assume when publishing to the central bus. The central bus permission remains owned by PRD-111 via the bus resource policy; this module must not rely on hidden edits in the Shared Services account to make spoke delivery work.

---

## 4. TERRAFORM SPECIFICATION

### Module Path
`connect-pbx/modules/l11-event-mesh/`

```hcl
# main.tf — Apply in each workload (spoke) account.

resource "aws_cloudwatch_event_rule" "forward_to_central" {
  count          = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  name           = "${var.org_name}-forward-to-central-${terraform.workspace}"
  event_bus_name = local.event_bus_name
  description    = "Forwards all connect-pbx events to central shared-services bus"

  event_pattern = jsonencode({
    source = [{ prefix = "connect-pbx" }]
  })
}

resource "aws_cloudwatch_event_target" "central_bus" {
  count          = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  rule           = aws_cloudwatch_event_rule.forward_to_central[0].name
  event_bus_name = local.event_bus_name
  target_id      = "central-event-bus"
  arn            = var.central_event_bus_arn
  role_arn       = aws_iam_role.event_bus_cross_account[0].arn

  dead_letter_config { arn = local.eventbridge_dlq_arn }
}

resource "aws_iam_role" "event_bus_cross_account" {
  count = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  name  = "${var.org_name}-eventbridge-cross-account-${terraform.workspace}"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "events.amazonaws.com" }
      Action    = "sts:AssumeRole"
    }]
  })
}

resource "aws_iam_role_policy" "event_bus_cross_account" {
  count = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  role  = aws_iam_role.event_bus_cross_account[0].id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = "events:PutEvents"
      Resource = var.central_event_bus_arn
    }]
  })
}
```

# PRD-113 — Cross-Account IAM Role Framework

---

## 1. METADATA

| Field | Value |
|---|---|
| **PRD ID** | PRD-113 |
| **Version** | 1.2.0 |
| **Layer** | 11 — Multi-Account Topology |
| **Depends On** | PRD-110 (Organizations), PRD-111 (Shared Services) |
| **Conditional foundation** | Yes — conditional foundation for multi-account shared-services access patterns selected by the manifest |

---

## 2. CONTEXT & PURPOSE

Provisions the cross-account IAM roles that allow the Shared Services account to read selected spoke workload resources. Implements the least-privilege cross-account access model — the Shared Services account can never write to spoke account resources, only read. Only the roles needed by manifest-enabled central services should be provisioned, and lower-layer resource ARNs must be passed in explicitly rather than inferred from hidden remote-state or source-edit assumptions. Shared Services consumers assume the spoke-side read roles directly; this PRD does not require a second relay role in the hub account.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — Spoke Account Readable Role
In each spoke account, provision IAM role `{org_name}-shared-services-read-{environment}` with trust policy allowing the Shared Services account ID to assume it. Permissions are granted only for the enabled shared services that require them, using explicit input ARNs such as DynamoDB `GetItem`/`Query` on Contact State and Agent State tables when PRD-31 is enabled, CloudWatch `GetMetricData`, and Lambda `InvokeFunction` on CRM adapter functions when the CRM pack is enabled.

### FR-002 — Direct Assume-Role Consumption
Shared Services Lambda functions and operators assume the spoke-side `{org_name}-shared-services-read-{environment}` roles directly by using their own execution role or operator credentials. No second hub-side relay role is required in this PRD.

---

## 4. TERRAFORM SPECIFICATION

### Module Path
`connect-pbx/modules/l11-cross-account-iam/`

```hcl
# main.tf — Apply in spoke workload accounts.
# Lower-layer resources are passed in as explicit inputs; this module does not discover them implicitly.

resource "aws_iam_role" "shared_services_read" {
  count = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  name  = "${var.org_name}-shared-services-read-${terraform.workspace}"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { AWS = "arn:aws:iam::${var.shared_services_account_id}:root" }
      Action    = "sts:AssumeRole"
      Condition = { StringEquals = { "sts:ExternalId" = var.cross_account_external_id } }
    }]
  })
}

resource "aws_iam_role_policy" "shared_services_read" {
  count = var.deployment_profile.account_topology == "spoke" ? 1 : 0
  role  = aws_iam_role.shared_services_read[0].id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid    = "DynamoDBRead"
        Effect = "Allow"
        Action = ["dynamodb:GetItem", "dynamodb:Query"]
        Resource = compact([
          var.contact_state_table_arn,
          var.agent_state_table_arn,
          var.contact_state_table_arn != "" ? "${var.contact_state_table_arn}/index/*" : "",
          var.agent_state_table_arn != "" ? "${var.agent_state_table_arn}/index/*" : ""
        ])
      },
      {
        Sid      = "CloudWatchRead"
        Effect   = "Allow"
        Action   = ["cloudwatch:GetMetricData", "cloudwatch:ListMetrics"]
        Resource = "*"
      }
    ]
  })
}
```

## 2. CONTEXT & PURPOSE

Aggregates SOC 2 and PCI evidence from all workload accounts into the Shared Services account. Provisions the organization-level Config aggregator and the cross-account evidence export Lambda that produces a single audit artifact covering all accounts for auditor review. If Security Hub delegated administration is enabled separately, its summaries may be one of the evidence inputs, but this PRD does not own Security Hub organization onboarding. This is a manifest-selected enterprise governance feature for organizations that want centralized evidence collection, not a mandatory baseline for every multi-account deployment.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — AWS Config Organization Aggregator
Provision `aws_config_configuration_aggregator` with `organization_aggregation_source` pulling Config data from all accounts in the organization. Deployed in the Shared Services account only after AWS Config delegated administration has been registered there or the module is applied from the management account.

### FR-002 — Cross-Account Evidence Export Lambda
Provision Lambda `{org_name}-org-evidence-export-{environment}` in the Shared Services account. Runs weekly (Monday 06:30 UTC — 30 minutes after per-account evidence exports). Aggregates evidence from all spoke accounts by assuming their shared-services-read roles and collecting their per-account evidence summaries from explicit account and bucket inputs. Writes a combined multi-account evidence summary to `s3://{central_audit_bucket}/evidence/org-weekly/{date}/org-summary.json`.

# PRD-115 — Account Vending Machine

---

## 1. METADATA

| Field | Value |
|---|---|
| **PRD ID** | PRD-115 |
| **Version** | 1.3.0 |
| **Layer** | 11 — Multi-Account Topology |
| **Depends On** | PRD-110 (Organizations), PRD-111 (Shared Services), PRD-113 (Cross-Account IAM) |
| **Blocks** | None — last PRD in Layer 11 |
| **Conditional foundation** | Yes — conditional foundation for account vending workflows selected by the manifest |

---

## 2. CONTEXT & PURPOSE

The Account Vending Machine (AVM) automates the provisioning of new AWS accounts into the organization. When a new regional office, business unit, or customer deployment needs a Connect instance, the AVM creates the member account, moves it into the correct OU, records its lifecycle in a registry, and dispatches approved bootstrap or baseline automation. It does not claim to run Terraform directly inside the new account from an in-process Lambda. This PRD implements the AVM as a Step Functions state machine. It is a conditional foundation even within multi-account topologies and is most appropriate when account creation is frequent enough to justify dedicated automation. Activation is governed by the module catalog and manifest, not by `deployment_profile`.

---

## 3. FUNCTIONAL REQUIREMENTS

### FR-001 — AVM Step Functions State Machine
Provision `aws_sfn_state_machine` named `{org_name}-account-vending-machine` with the following states:

1. **CreateAccount** — `aws organizations create-account`
2. **WaitForAccount** — Poll until account status is ACTIVE
3. **MoveToOU** — Move account to correct OU (Dev/Staging/Prod based on input)
4. **RegisterSpoke** — Seed or update the account registry and any additive Shared Services registrations that are explicitly owned by this layer
5. **StartBaselineExecution** — Start the approved bootstrap or baseline execution path (for example a CodeBuild or workflow dispatch) with explicit inputs; do not run Terraform CLI directly inside the AVM Lambda
6. **NotifyComplete** — SNS notification to operations team with the new account details and the dispatched bootstrap execution ID

### FR-002 — Account Bootstrap Lambda
Provision Lambda `{org_name}-account-bootstrapper-{environment}` that validates the newly created account metadata, updates the registry, and dispatches the approved baseline or bootstrap execution path using explicit project or workflow inputs. It must not embed the Terraform CLI, modify repo source, or claim baseline success before the downstream execution reports completion.

### FR-003 — Account Registry Table
Provision DynamoDB table `{org_name}-account-registry-{environment}` in the Shared Services account tracking all vended accounts:

| Attribute | Type | Description |
|---|---|---|
| `AccountId` | S (PK) | AWS account ID |
| `AccountName` | S | Human-readable name |
| `OUPath` | S | OU path (Dev/Staging/Prod) |
| `Status` | S | PROVISIONING / ACCOUNT_CREATED / BASELINE_IN_PROGRESS / ACTIVE / FAILED / DECOMMISSIONED |
| `VendedAt` | S | ISO 8601 |
| `ConnectInstanceId` | S | Populated after Layer 1 apply |
| `BaselineExecutionId` | S | CodeBuild build ID or workflow execution ID for the dispatched baseline |
| `LastError` | S | Most recent provisioning or bootstrap failure detail |
| `IdempotencyKey` | S | Request key used to make CreateAccount retries safe |
